# Zigns Tizen Player Spike

Technical spike for a Samsung Tizen TV / Smart Signage Web Application wrapper
around the hosted Zigns display player.

## Status

- Current state: package-ready source scaffold plus Bash and PowerShell spike
  scripts.
- Package target: `.wgt` Web application built with Tizen Studio and the TV
  extension.
- Windows packaging/signing on 2026-05-05 succeeded with Tizen Studio installed
  at `C:\tizen-studio`, `profiles.xml` at
  `C:\tizen-studio-data\profile\profiles.xml`, and Samsung profile
  `zigns-tv-dev`.
- Not production-supported until it is packaged, signed, installed, and smoke
  tested on real Samsung commercial signage hardware.

## Wrapper Design

The packaged app launches `index.html`, which immediately redirects to:

```text
https://app.zigns.io/display.html?nativePlatform=tizen&nativeVersion=0.1.0
```

The hosted player remains the source of truth for pairing, screen-token auth,
playlist playback, offline cache, analytics, and diagnostics. The local Tizen
package exists to satisfy TV installation, launcher, signing, and future
platform-specific hooks.

## Build Requirements

- Tizen Studio with Samsung TV extension.
- Certificate Manager and Samsung Certificate Extension.
- A Samsung certificate profile suitable for the target device or store path.
- A target TV/signage panel with Developer Mode enabled, or a Tizen TV emulator.
- PATH entry for the Tizen CLI, typically `<TIZEN_STUDIO>/tools/ide/bin`.
- Optional PATH entry for `sdb` so target discovery is available from the shell.

Windows setup notes from the first successful local package:

- Package Manager's Main SDK repository may need `https` instead of `http`:
  `https://usa.sdk-dl.tizen.org/sdk/tizenstudio`.
- Install `Certificate Manager` from the Main SDK tab, then install the Samsung
  TV extension and `Samsung Certificate Extension`.
- Prefer a Samsung certificate profile for TV packages, not the generic Tizen
  profile. The local profile used for the spike was `zigns-tv-dev`.
- The Samsung TV emulator path can fail on Windows with `intelhaxm` / HAXM when
  Hyper-V, WSL2, or Android Studio virtualization is enabled. Emulator support
  is optional for Zigns; real-device install validation is the production gate.

## Signing Inputs

The package must be signed before it can be installed on a TV or submitted
through Samsung distribution. The local script expects a security profile that
already exists in Tizen Studio.

Environment variables:

```bash
export TIZEN_PROFILE="zigns-tv-dev"              # Tizen security profile name
export TIZEN_PROFILES_XML="$HOME/tizen-studio-data/profile/profiles.xml"
export TIZEN_TARGET="SamsungTV"                  # optional install target
```

`TIZEN_PROFILES_XML` is optional if the Tizen CLI already knows the profiles
path. `TIZEN_TARGET` is optional; omit it to build only.

On Windows/PowerShell:

```powershell
$env:TIZEN_PROFILE = "zigns-tv-dev"
$env:TIZEN_PROFILES_XML = "C:\tizen-studio-data\profile\profiles.xml"
```

Do not commit `profiles.xml`, `.p12`, `.pwd`, keystores, distributor
certificates, or anything under `C:\Users\<you>\SamsungCertificate`.

## Package / Install Sketch

From the repo root:

```bash
bash player-tizen/package-spike.sh
```

From Windows PowerShell, prefer the Windows-local staging script:

```powershell
.\player-tizen\package-spike.ps1
```

Default output:

```text
C:\Users\<you>\zigns-tizen-build\.buildResult\Zigns Player.wgt
```

The PowerShell script intentionally stages source files into a local Windows
directory before calling `tizen build-web`. This avoids Tizen CLI resolving a
`\\wsl$\...` working path as `C:\Windows\...`.

The script performs the expected CLI flow:

```bash
node scripts/generate-player-icons.js
tizen build-web -- .
tizen package -t wgt -s <security-profile> -- .buildResult
sdb devices
tizen install -n ZignsPlayer.wgt -t <device-name>
```

If the Bash script cannot find `tizen`, it stops before modifying package output
and prints the missing-tooling gate. The Bash script stages only `config.xml`,
`index.html`, and `icon.png` into a temporary directory so README/scripts are
not bundled into the TV package.

To install later when a TV target is available:

```powershell
.\player-tizen\package-spike.ps1 -Target "SamsungTV" -Install
```

## Package Artifacts

- `icon.svg` remains the source icon.
- `icon.png` is generated by `scripts/generate-player-icons.js` and referenced
  by `config.xml`.
- `.wgt`, `.buildResult/`, `.metadata/`, and signing folders are ignored by Git.
- Local staging directories such as `player-tizen-build/` and
  `C:\Users\<you>\zigns-tizen-build` are generated scratch output.

## Spike Validation Checklist

- `.wgt` builds and signs with a local Samsung certificate profile.
- App installs and launches on a Samsung TV/signage device.
- First launch shows the Zigns pairing code.
- Admin pairing succeeds and survives app restart.
- Firestore realtime updates work for publish, delete, and replace operations.
- Image, video, PDF, designed slide, webpage, and YouTube slides render.
- YouTube autoplay and iframe behavior are acceptable on target firmware.
- Offline cache behavior matches Chrome/Android expectations.
- Remote-control keys do not expose unwanted browser chrome.
- Device power-on or signage scheduler can relaunch the app unattended.
- Admin diagnostics reports shell `tizen 0.1.0`, viewport, media support, and
  online/offline transitions.

## Open Questions

- Minimum supported Tizen version and panel model year.
- Best unattended launch path for Samsung commercial signage: native app,
  MagicINFO, URL Launcher fallback, or partner/store distribution.
- Whether strict content-security or access-origin rules need tightening after
  the media/CDN/widget matrix is known.
- Whether a local Tizen service is needed for watchdog or boot behavior.
